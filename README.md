## вҡЎ Quick Start

рҹҢҗ **Just want to explore?** Browse community skills, evolution lineage at **[open-space.cloud](https://open-space.cloud)** вҖ” no installation needed.

```bash
git clone https://github.com/HKUDS/OpenSpace.git && cd OpenSpace
pip install -e .
openspace-mcp --help   # verify installation
```

> [!TIP]
> **Codex MCP deployment:** If you want this repository to provide global
> `openspace` and `openspace_evolution` MCP servers for a fresh Codex setup,
> follow [`docs/codex-mcp-deployment.md`](docs/codex-mcp-deployment.md).

> [!TIP]
> **Recommended split routing for OpenAI-compatible gateways**
>
> If your main model runs through an OpenAI-compatible provider or local relay (for example `gpt-5.4` via `http://127.0.0.1:8080/v1`), the recommended default is:
> - keep the main LLM on that provider via `OPENSPACE_LLM_*`
> - keep skill-router embeddings local via `OPENSPACE_SKILL_EMBEDDING_BACKEND=local`
> - use `OPENSPACE_SKILL_EMBEDDING_MODEL=BAAI/bge-small-en-v1.5`
>
> Why this is the default recommendation:
> - lower latency for routing and prefilter
> - no dependence on a remote `/v1/embeddings` endpoint
> - no extra token spend for embedding generation
> - stronger main LLM still handles final reasoning and selection
>
> Architecture notes and flow diagram: [`docs/current-routing-flow.md`](docs/current-routing-flow.md)

> [!TIP]
> **Slow clone?** The `assets/` folder (~50 MB of images) makes the default clone large. Use this lightweight alternative to skip it:
> ```bash
> git clone --filter=blob:none --sparse https://github.com/HKUDS/OpenSpace.git
> cd OpenSpace
> git sparse-checkout set '/*' '!assets/'
> pip install -e .
> ```

### рҹӨ– Path A: For Your Agent

Works with any agent that supports skills (`SKILL.md`) вҖ” [Claude Code](https://docs.anthropic.com/en/docs/claude-code), [Codex](https://github.com/openai/codex), [OpenClaw](https://github.com/openclaw/openclaw), [nanobot](https://github.com/HKUDS/nanobot), etc.

**в‘  Add OpenSpace to your agent's MCP config:**

```json
{
  "mcpServers": {
    "openspace": {
      "command": "openspace-mcp",
      "toolTimeout": 600,
      "env": {
        "OPENSPACE_HOST_SKILL_DIRS": "/path/to/your/agent/skills",
        "OPENSPACE_WORKSPACE": "/path/to/OpenSpace",
        "OPENSPACE_API_KEY": "sk-xxx (optional, for cloud)"
      }
    }
  }
}
```

> [!TIP]
> Credentials (API key, model) are **auto-detected** from your agent's config; you usually don't need to set them manually.

**в‘Ў Copy skills** into your agent's skills directory:

```bash
cp -r OpenSpace/openspace/host_skills/delegate-task/ /path/to/your/agent/skills/
cp -r OpenSpace/openspace/host_skills/skill-discovery/ /path/to/your/agent/skills/
```

Done. These two skills teach your agent when and how to use OpenSpace вҖ” no additional prompting needed. Your agent can now self-evolve skills, execute complex tasks, and access the cloud skill community. You can also add your own custom skills вҖ” see [`openspace/skills/README.md`](openspace/skills/README.md).

> [!NOTE]
> **Cloud community (optional):** Register at **[open-space.cloud](https://open-space.cloud)** to get a `OPENSPACE_API_KEY`, then add it to the `env` block above. Without it, all local capabilities (task execution, evolution, local skill search) work normally.

рҹ“– Per-agent config (OpenClaw / nanobot), all env vars, advanced settings: [`openspace/host_skills/README.md`](openspace/host_skills/README.md)

### рҹ‘Ө Path B: As Your Co-Worker

Use OpenSpace directly вҖ” coding, search, tool use, and more вҖ” with self-evolving skills and cloud community built in.

> [!NOTE]


### рҹ§¬ Self-Evolution Engine

The core of OpenSpace. Skills aren't static files вҖ” they're living entities that automatically select, apply, monitor, analyze, and evolve themselves.

#### рҹ”„ Autonomous & Continuous Evolution

- **Full Lifecycle Management**: From discovery to application to evolution вҖ” all without human intervention. OpenSpace completes tasks regardless of whether matching skills exist.

**Three Evolution Modes**:
- рҹ”§ FIX вҖ” Repair broken or outdated instructions in-place. Same skill, new version.
- рҹҡҖ DERIVED вҖ” Create enhanced or specialized versions from parent skills. New skill directory, coexists with parents.
- вңЁ CAPTURED вҖ” Extract novel reusable patterns from successful executions. Brand new skill, no parent.

**Three Independent Triggers**: Multiple lines of defense against skill degradation вҖ” both successful and failed executions drive evolution.
- **рҹ“Ҳ Post-Execution Analysis** вҖ” Runs after every task. Analyzes full recordings and suggests FIX/DERIVED/CAPTURED for involved skills.
- **вҡ пёҸ Tool Degradation** вҖ” When tool success rates drop, quality monitor finds all dependent skills and batch-evolves them.
- **рҹ“Ҡ Metric Monitor** вҖ” Periodically scans skill health metrics (applied rate, completion rate, fallback rate) and evolves underperformers.

#### рҹ“Ҡ Full-Stack Quality Monitoring
Multi-Layer Tracking: Quality monitoring covers the entire execution stack вҖ” from high-level workflows to individual tool calls:
- **рҹҺҜ Skills** вҖ” applied rate, completion rate, effective rate, fallback rate
- **рҹ”Ё Tool Calls** вҖ” success rate, latency, flagged issues
- **вҡЎ Code Execution** вҖ” execution status, error patterns

**Cascade Evolution**: When any component degrades вҖ” skill workflow or single tool call вҖ” evolution automatically triggers for all upstream dependent skills, maintaining system-wide coherence.

#### рҹ”§ Intelligent & Safe Evolution
**рҹӨ– Autonomous Evolution**: Each evolution explores the codebase, discovers root causes, and decides fixes autonomously вҖ” gathering real evidence before making changes, not generating blindly.

**вҡЎ Diff-Based & Token-Efficient**: Produces minimal, targeted diffs rather than full rewrites, with automatic retry on failure. Every version stored in a version DAG with full lineage tracking.

**рҹӣЎпёҸ Built-in Safeguards**:
- Confirmation gates reduce false-positive triggers
- Anti-loop guards prevent runaway evolution cycles
- Safety checks flag dangerous patterns (prompt injection, credential exfiltration)
- Evolved skills are validated before replacing predecessors

**рҹҢҗ Collaborative Skill Community**
A collaborative registry where agents share evolved skills. When one agent evolves an improvement, every connected agent can discover, import, and build on it вҖ” turning individual progress into collective intelligence.

- **рҹ”җ Flexible Sharing**: Share skills publicly, within groups, or keep them private. Smart search finds what you need and auto-imports it. Every evolution is lineage-tracked with full diffs.

- **вҳҒпёҸ Collaborative Platform**: open-space.cloud вҖ” register for an API key, browse community skills, and manage your groups.

## рҹӨқ Contribute & Roadmap

We welcome contributions! OpenSpace today evolves *how to do X*. The next frontier: **evolving how agents organize doing X together**. 

Group infrastructure (visibility, sharing, permissions) is already live. What comes next:

- [ ] **[Kanban](https://github.com/BloopAI/vibe-kanban)-style orchestration** вҖ” Shared task board with skill-aware scheduling; scheduling itself evolves
- [ ] **Collaboration pattern evolution** вҖ” Decomposition, handoff, prioritization strategies captured and improved from completed tasks
- [ ] **Role emergence** вҖ” Agents develop role profiles through practice, not configuration
- [ ] **Cross-group pattern transfer** вҖ” Coordination patterns discovered by one group available to others via cloud registry
